OpenFlight API Release Notes

Last revised: 02/05/16

This section contains the release notes for all versions of the OpenFlight API beginning with v2.1 and ending with the current version. The release notes are listed here in reverse chronological order (i.e., the most recent version first). In addition to version specific release notes, there is also a general release notes section here containing information applicable to the most current version.

General

The OpenFlight API is distributed in the following binary formats for the Windows platform:

If you are developing plug-ins for Creator 15, you should install and use VC12-Win32 or VC12-x64 depending on which version of Creator you are using (32 or 64 bit, respectively) so your plug-ins are compatible.

If you are developing plug-ins for Creator 14, you should install and use VC9-Win32 so your plug-ins are compatible.

When you install the OpenFlight API, you can choose which binary format you want or you can choose to install them all. In this way, you can have multiple binary formats installed on your computer simultaneously and then choose (at runtime) which format to run with.

Installed Directories

On both Windows and Linux platforms, the OpenFlight API is installed into the Presagis Common Directory Organization. The OpenFlight API files are distributed into several subfolders under the root of this organization. This root folder is different depending on the platform but the subfolder structure below this root is the same for all platforms.

By convention, the environment variable PRESAGIS_OPENFLIGHT_API is used to specify the root installation folder of the OpenFlight API. Additionally, environment variables of the form PRESAGIS_OPENFLIGHT_API_version where version specifies the major and minor version numbers of the distribution, are used to maintain multiple versions of the OpenFlight API simultaneously. For example, the environment variable PRESAGIS_OPENFLIGHT_API_15_0 would correspond to OpenFlight API version 15.

The Windows installer creates these two environment variables automatically.

The Linux installer includes a tcsh script file that you can execute to set these environment variables manually.

The following sections describe the subfolders below the OpenFlight installation root folder. Depending on which platform and compiler/architecture you choose to install, <platform> in the following sections will be:

Plug-in Runtime Directory

Plug-ins are loaded in the stand-alone program environment when your program or script calls the OpenFlight API function mgInit. mgInit searches for plug-ins in the following locations (in the order listed). As soon as a folder is found in one of these locations, mgInit stops searching for other locations and loads the plug-ins it finds in the first folder found.

  1. The directory (or directories) your program specifies when calling mgSetPluginFolder. If you call this function before mgInit to specify one or more directories, the folder(s) you specify will be searched for plug-ins. To specify more than one directory, pass a semi-colon delimited list of paths to mgSetPluginFolder.

  2. The directory (or directories) specified by the environmental variable PRESAGIS_OPENFLIGHT_PLUGIN_DIR. This environmental variable may specify any accessible directory on your computer or may specify multiple directories using a semi-colon to delimit each path.

  3. A directory named plugins immediately below the directory that contains the executable being run.

Creator loads plug-ins automatically when it starts. Creator searches for plug-ins in the following locations (in the order listed). As soon as a folder is found in one of these locations, Creator stops searching for other locations and loads the plug-ins it finds in the first folder found.

  1. The directory (or directories) specified by the environmental variable PRESAGIS_CREATOR_PLUGIN_DIR. This environmental variable may specify any accessible directory on your computer or may specify multiple directories using a semi-colon to delimit each path.

  2. A directory named plugins in the Creator configuration folder. This folder is located at:

    PRESAGIS_ROOT/Creator/config/plugins

Visual Studio Project Settings

It is very important that your plug-in module be linked with the correct system run-time libraries to match Creator. In particular, multithreaded DLLs are required. In Visual Studio, open the Project Properties window, select the C/C++ tab and then select the Code Generation category. For Runtime Library, select Multi-threaded Debug DLL for the Win32 Debug configuration and Multi-threaded DLL for the Win32 Release configuration.

Deprecated API

Please review the list of deprecated API symbols and functions that appear in the Deprecated API section of this reference document. For each symbol that appears in this section, an appropriate replacement symbol or strategy is also given. Although deprecated API symbols are still supported in the current version of the API, they may not be in future releases. To ensure that code you develop using the MultiGen OpenFlight API remains compatible with future versions of the API, you should eliminate any deprecated symbols from your code.

When you compile code that uses a deprecated symbol, the compiler may issue a warning message stating that the symbol is not defined or has been deprecated. This is expected and is an indication to you that action should be taken to replace that symbol as described above. Note that even though the compiler issues this warning message, your program or plug-in should link successfully even when it uses a deprecated symbol.

v14.0

This version of the OpenFlight API corresponds to OpenFlight version 16.4.

Loading plug-ins

Loading plug-ins in stand-alone programs

The way in which plug-ins are located by stand-alone programs has changed.

A new function, mgSetPluginFolder, was added that allows you to specify the directory (or directories) from which plug-ins will be loaded into your stand-alone program. You must call this function before mgInit. When you do, the folder(s) you specify will be searched for plug-ins by mgInit. In this way, you can control exactly which plug-ins you want to load into your stand-alone program environment. To specify more than one directory, pass a semi-colon delimited list of paths to mgSetPluginFolder.

Stand-alone programs no longer use the PRESAGIS_CREATOR_PLUGIN_DIR environment variable to locate plug-ins. Instead they use the environment variable PRESAGIS_OPENFLIGHT_PLUGIN_DIR. This allows more control on systems where both Creator and stand-alone applications are used simultaneously. Furthermore, the environment variable PRESAGIS_OPENFLIGHT_PLUGIN_DIR may now specify multiple directories, each delimited by a semi-colon.

Loading plug-ins in Creator

The way in which plug-ins are located by Creator has changed.

Creator still uses the PRESAGIS_CREATOR_PLUGIN_DIR environment variable to locate plug-ins. However, the value of this environment variable may now specify multiple directories, each delimited by a semi-colon.

Creator no longer searches for a folder named plugins immediately below the directory that contains the Creator executable. When an alternative plug-in folder is desired, use the environment variable PRESAGIS_CREATOR_PLUGIN_DIR.

Light Point API Compatibility

In version 2.6 of the OpenFlight API, the attributes of light point nodes (fltLightPoint ) were "palette-ized". This introduced a level of indirection on the light node (similar to color, material and texture on fltPolygon and fltMesh nodes) into two separate palette entries, the Light Point Appearance Palette and Light Point Animation Palette, fltLpAppearancePalette and fltLpAnimationPalette , respectively.

Regrettably, this "broke" existing applications that use mgGetAttList and mgSetAttList to access attributes directly on light point nodes. In order to function correctly, such applications had to be updated to first retrieve the appropriate light point palette record referenced by a light point node and then use mgGetAttList and mgSetAttList on that palette record rather than the light point node directly.

In this version of the OpenFlight API, this situation has been addressed by enhancing mgGetAttList and mgSetAttList to allow access to light point palette attributes directly from the light point node. Specifically, if you now call mgSetAttList on a light point node specifying any attribute from the Light Point Appearance Palette or Light Point Animation Palette, the corresponding attribute of the palette entry referenced by the light point node will be set. Similarly, if you use mgGetAttList on a light point node to get a light point palette attribute, the attribute of the palette entry referenced by the light point node will be retrieved.

Note that if you call mgGetAttList or mgSetAttList on a light point node to access a palette attribute and the light point node does not yet have a palette entry associated, a new palette entry will be created, the corresponding index on the light point node will be updated to reference this new palette entry and this new palette entry will be accessed.

Plug-in Runtime Directory

The directory specified by the environment variable MGPLUGINDIR supersedes all other plug-in runtime directories searched. Additionally, once a plug-in runtime directory is located, it and all subdirectories below it are recursively searched for plug-in modules. See the OpenFlight API Developer Guide, Volume 2 for more information.

Automatic Version Update for Data Extension Records

Padding is no longer required to reserve space for future additions to struct definitions. When the OpenFlight API finds an extension record in a database file that is shorter than the current corresponding struct definition, the extension record (when it is read into memory) is automatically expanded to accommodate for the new fields at the end of the new struct definition. When this happens, the values for these new fields are initialized to 0. See the OpenFlight API Developer Guide, Volume 2 for more information.
